# AgriAdvisoryFulfillmentAttributes Schema

**Container:** `Fulfillment.beckn:deliveryAttributes`
**Version:** 1.0.0
**Use Cases:** Crop/Disease Advisory (UC1), Commodity Price Listing (UC2), Weather Forecast Data (UC3)
**Tag:** agri-fulfillment agri-digital-delivery agri-advisory

---

## Overview

`AgriAdvisoryFulfillmentAttributes` is the fulfillment-level extension schema shared
across all three Agri Advisory use cases on the Beckn v2 network. It extends
`beckn:Fulfillment.beckn:deliveryAttributes` for `mode: DIGITAL` fulfillments.

The schema covers:
- How digital content is delivered (`deliveryMode`)
- The geographic target for location-specific services (`targetLocation`, `targetRegionRadiusKm`)
- Post-purchase content access details (`contentAccessUrl`, `contentAccessExpiry`)
- The language in which the content is to be delivered (`requestedLanguage`)
- Price query temporal scope for UC2 (`priceQueryTimeRange`)

All three use cases produce DIGITAL fulfillments â€” no physical logistics are involved.

## Design Rationale

### Single shared schema for all three UCs
The fulfillment semantics across UC1, UC2, and UC3 are structurally identical: all involve
locating a digital content asset and delivering it to a buyer digitally. The key variable
attributes (delivery mode, target location, delivery language) are the same properties
across all three UCs. A shared schema avoids duplication and reduces the number of
schema packs a BPP needs to implement.

UC-specific fulfillment concerns:
- UC1 (crop advisory): `targetLocation` carries farm GPS for pest forecast; `deliveryMode`
  carries the format (video link, PDF, etc.)
- UC2 (price listing): `targetLocation` + `targetRegionRadiusKm` define the mandi search
  area; `priceQueryTimeRange` defines the date window
- UC3 (weather forecast): `targetLocation` carries farm GPS; `contentAccessUrl` carries
  the PDF report link post-order

### targetLocation in fulfillmentAttributes, not itemAttributes
In v1, the search location was in `intent.fulfillment.stops[].location.gps`. This is
execution metadata (WHERE to apply the service), not intrinsic item metadata (WHAT the
service IS). In v2, execution semantics belong in `Fulfillment.deliveryAttributes`, not
in `itemAttributes`. This separation allows the same item (e.g., a hyperlocal weather
forecast product) to be fulfilled at any location.

### requestedLanguage in fulfillmentAttributes vs itemAttributes.languages
`itemAttributes.languages` lists all available languages for an item (discovery filter).
`requestedLanguage` in fulfillmentAttributes is the buyer's delivery preference â€” it
specifies which language the BPP should use when preparing the actual delivery. These
are two different concerns at two different lifecycle stages.

### contentAccessUrl as fulfillment attribute
The URL to access delivered content is execution-state information, not intrinsic item
metadata. It is generated by the BPP upon order confirmation and communicated via the
fulfillment response. It must not be in itemAttributes (which is catalog-time metadata).

### priceQueryTimeRange for UC2
In v1, the time range for price discovery was in `intent.fulfillment.stops[].time.range`.
In v2, this execution-time query parameter belongs in `fulfillmentAttributes` as it
defines the temporal scope of the fulfillment (which period's prices to return), not
the intrinsic identity of the item.

### PII: Farm GPS as sensitive data
`targetLocation` carries farm-level GPS coordinates. This is sensitive PII that identifies
a farmer's property. The `profile.json` flags this for privacy handling. BAP/BPP platforms
must implement consent flows and must not persist or share this data beyond the transaction.

## Non-Goals

- **Does not model physical delivery logistics** â€” all agri advisory fulfillments are DIGITAL.
- **Does not model multi-stage fulfillment** â€” all use cases confirmed as single-stage.
- **Does not model recurring fulfillments/subscriptions** â€” out of scope per requirements.
- **Does not model tracking** â€” not required per requirements.

## Upstream Candidates

| Attribute | Rationale |
|-----------|-----------|
| `deliveryMode` | Any digital content domain (e-learning, telemedicine, media) needs a delivery mode classification for `DIGITAL` fulfillments |
| `contentAccessUrl` | Universal for digital fulfillments; candidate for promotion to Beckn core DIGITAL fulfillment mode |
| `contentAccessExpiry` | Useful in any time-limited digital access scenario |
| `targetLocation` | For any location-sensitive service (weather, soil testing, satellite imagery), location in fulfillmentAttributes is the correct pattern |
